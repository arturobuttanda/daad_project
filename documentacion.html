Introducción

🛒 ¿Qué es NexusMarket?

NexusMarket es una plataforma de comercio electrónico con dos tipos de usuarios:

🧑‍💼 Vendedor

Puede crear, editar y eliminar productos de su inventario. También ve reportes de ventas, gráficas financieras y recibe recomendaciones de precio basadas en inteligencia artificial.

🛍️ Cliente

Navega el marketplace, ve el historial de precios de cada producto con gráficas tipo Keepa, agrega productos al carrito y realiza compras. Recibe análisis de conveniencia de compra.

ℹ️
¿Para qué sirve la recomendación de precio? Cuando un vendedor agrega un producto nuevo, el sistema analiza todos los productos similares ya existentes en el catálogo y sugiere automáticamente un precio competitivo usando inteligencia artificial (TF-IDF + Similitud Coseno).

Comunicación cliente-servidor

🌐 ¿Qué es una API REST?

📖
Analogía: un restaurante El cliente (navegador) es el comensal. La API REST es el mesero. La base de datos es la cocina. El comensal pide algo al mesero con un formato estándar (HTTP), el mesero lo lleva a la cocina y trae la respuesta en JSON.

Una API REST usa los métodos HTTP estándar:

MétodoAcciónEjemplo en NexusMarket
GETObtener datosGET /api/productos → lista de productos
POSTCrear nuevoPOST /api/productos → crear producto
PUTActualizarPUT /api/productos/PROD-001 → editar precio
DELETEEliminarDELETE /api/productos/PROD-001 → borrar

Lógica de negocio

⚡ Backend — FastAPI y el modelo orientado a objetos

¿Qué es FastAPI?

FastAPI es un framework Python para crear APIs web. Sus tres características clave para este proyecto son:

1️⃣ Validación automática (Pydantic)

Cuando el frontend envía JSON, FastAPI lo convierte automáticamente a un objeto Python y verifica que los tipos sean correctos. Si falta un campo obligatorio, devuelve error 422 automáticamente.

2️⃣ Documentación automática

Visitar http://localhost:8000/docs muestra la documentación interactiva Swagger de todos los endpoints. Se genera sola a partir del código.

3️⃣ CORS

Permite que el HTML (que está en un puerto diferente) pueda llamar a la API. Sin CORS configurado, el navegador bloquearía las peticiones por seguridad.

Los modelos de solicitud (Pydantic)

Cada endpoint que recibe datos tiene un modelo que define exactamente qué espera:

Inteligencia artificial clásica

🧠 Recomendador de precios — TF-IDF + Similitud Coseno

🎯
Objetivo Dado un producto nuevo (nombre + marca + categoría), encontrar los productos más parecidos en el catálogo y calcular un precio justo basado en ellos. Sin internet, sin GPU, sin API de terceros.

Paso 1 — ¿Qué es TF-IDF?

TF-IDF significa Term Frequency — Inverse Document Frequency. Es un algoritmo que convierte texto en números para poder comparar documentos matemáticamente.

TF (Term Frequency)

¿Con qué frecuencia aparece una palabra en este documento? Una palabra que aparece mucho en un texto es probablemente importante para ese texto.

Ejemplo: "televisor" aparece 3 veces → alta TF

IDF (Inverse Document Frequency)

¿En cuántos documentos aparece esa palabra? Una palabra que aparece en TODOS los documentos (como "el", "de") no es útil para diferenciarlos.

Ejemplo: "producto" aparece en todos → baja IDF → peso bajo

TF-IDF(t,d) = TF(t,d) × log(N / df(t))

Donde t = término, d = documento, N = total documentos, df(t) = documentos con ese término.

Paso 2 — Construcción del vector de cada producto

Cada producto se convierte en un "documento" de texto concatenando sus campos:

def _construir_documento(self, producto):
    nombre    = normalizar(producto["nombre"])    # "televisor samsung"
    marca     = normalizar(producto["marca"])     # "samsung"
    categoria = normalizar(producto["categoria"]) # "electronica"
    return f"{nombre} {marca} {categoria}"
    # → "televisor samsung samsung electronica"

Paso 3 — Matriz TF-IDF del catálogo

# scikit-learn construye la matriz automáticamente
vectorizador = TfidfVectorizer(
    ngram_range=(1, 2),      # Considera palabras solas Y pares: "televisor samsung"
    sublinear_tf=True,     # Aplica log al TF para suavizar frecuencias altas
)
matriz = vectorizador.fit_transform(documentos)
# Resultado: matriz (n_productos × n_términos) con valores TF-IDF

Paso 4 — Similitud Coseno

Para comparar dos vectores, se calcula el coseno del ángulo entre ellos. Si dos productos tienen las mismas palabras en proporciones similares, el ángulo es pequeño (coseno cercano a 1).

similitud(A, B) = (A · B) / (|A| × |B|)

El resultado va de 0 (sin relación) a 1 (idénticos). Se usa sklearn.metrics.pairwise.cosine_similarity.

Paso 5 — Filtrado y precio ponderado

# Solo se conservan productos con similitud ≥ 0.75
similares = [s for s in resultados if s.similitud >= 0.75][:20]

# Precio recomendado = promedio ponderado por similitud
precio = Σ(precio_i × similitud_i) / Σ(similitud_i)
💡
¿Por qué promedio ponderado y no promedio simple? Un producto con similitud 0.98 (casi idéntico) debería influir mucho más en el precio final que uno con similitud 0.76. El promedio simple trataría ambos igual. El promedio ponderado da más peso a los más parecidos.

Paso 6 — Caché inteligente (Singleton + firma)

def _asegurar_indice(self):
    # Obtiene conteo_productos para invalidar caché del recomendador
    firma = db.obtener_firma_catalogo_similitud()

    if self._firma == firma and self._vectorizador is not None:
        return  # El catálogo no cambió → reutilizar índice

    # El catálogo cambió → reentrenar TF-IDF
    catalogo = db.cargar_catalogo_similitud(firma)
    self._ajustar_vectorizador(catalogo)
    self._firma = firma

El frontend es HTML puro con JavaScript moderno (ES6+). No usa ningún framework de componentes. Cada página es un archivo .html independiente que, al cargarse, llama al backend con fetch() para obtener datos.

¿Cómo se maneja la sesión?

No hay tokens JWT. Al hacer login, el backend devuelve el objeto usuario (id, nombre, tipo) y el frontend lo guarda en localStorage. Cada página verifica al cargar si hay usuario y si su tipo coincide:

const usuario = obtenerUsuario();
if (!usuario.registrado || usuario.tipo !== "Vendedor") {
    window.location.href = "../iniciar-sesion.html";
}

🔄 Flujos completos del sistema

Flujo 1 — Vendedor crea un producto y recibe precio sugerido

1 Vendedor escribe nombre, marca, categoría en el formulario
2 JavaScript espera 600ms (debounce) y llama POST /api/productos/recomendacion-precio
3 FastAPI recibe la solicitud y llama RecomendadorPrecio.recomendar()
4 _asegurar_indice() verifica si el catálogo cambió. Si sí, recarga y reentrena TF-IDF
5 Se construye el documento del producto objetivo y se calcula coseno contra toda la matriz
6 Se filtran similitudes ≥ 0.75, se calcula promedio ponderado y se retorna JSON
7 JavaScript muestra precio recomendado, similitud prom/máx, producto más similar y tabla
8 Vendedor hace clic en "Guardar" → POST /api/productos → Oracle INSERT

Flujo 2 — Cliente compra un producto

1 Cliente navega el marketplace → GET /api/cliente/productos
2 Hace clic en un producto → navega a producto.html?id=PROD-001
3 GET /api/cliente/productos/PROD-001 devuelve: producto + historial + análisis de compra
4 JavaScript dibuja la gráfica de precios con Chart.js y calcula la recomendación de compra
5 Cliente selecciona cantidad y hace clic en "Comprar ahora"
6 POST /api/cliente/compras → Oracle descuenta stock, inserta venta y detalles
7 Se muestra el ticket de compra en pantalla con el número de transacción

Flujo 3 — Inicio de sesión

1 Usuario escribe correo + contraseña → POST /api/auth/login
2 FastAPI consulta Oracle: SELECT password_hash FROM usuarios WHERE correo = ?
3 passlib.verify(contrasena_ingresada, hash_guardado) — verifica la contraseña con PBKDF2
4 Si es válida: devuelve {id, nombre, tipo_usuario}
5 JavaScript guarda en localStorage y redirige a inventario.html o marketplace.html

¿Cómo arranca el servidor?

# Desde la raíz del proyecto:
python run.py

# run.py ejecuta internamente:
uvicorn Backend.app:app --host 127.0.0.1 --port 8000 --reload

uvicorn es el servidor web ASGI que ejecuta FastAPI. El flag --reload reinicia automáticamente al detectar cambios en el código.

🔧
Variables de entorno (.env) El archivo .env en la raíz contiene las credenciales de Oracle (DB_USER, DB_PASSWORD, DB_DSN, WALLET_PATH). Nunca se sube a git. La librería python-dotenv lo carga automáticamente al iniciar el servidor.
Resumen

✅ Resumen de la arquitectura

HTML + JS puro

Solo muestra datos y llama a la API. No contiene lógica de negocio. Usa fetch() para comunicarse.

FastAPI (app.py)

Define las rutas. Valida datos de entrada. Delega a servicios y responde JSON al frontend.

modelo_poo.py

Define las entidades del negocio (Persona, Producto, Venta, Informe) con sus reglas y validaciones.

recomendador_precio.py

Clase RecomendadorPrecio: TF-IDF + Coseno. Toda la IA vive aquí, separada del resto.

conexion_base.py

Única capa que toca Oracle. Contiene todas las consultas SQL. Maneja transacciones.

Oracle Cloud

Almacena todos los datos. Se conecta por TLS con wallet. No se puede modificar la estructura.
